  It only does alpha-beta up to a certain depth, and that depth was 2
by default -- i.e., not far at all. At the leaves at that depth, it
looks for the "one arch left" condition that it knows is a win, but
that's pretty much all it has to distinguish between good and bad
positions. So most of the time it's moving "randomly," with no way to
tell whether it's winning or losing. Like I said, alpha-beta /really/
fails here.

If it's really true that
the first player can always win,

  It's not, in general. Trivially, the 2x2 board is a second-player
win (as is the 1x1! :) The 3x3 is a first-player win. I think the 4x4
is a second-player win.

it might be worth experimenting with
rule changes to come up with a rule set which is "more even" (e.g. which
favors player 2 for some sizes of board, which would perhaps suggest that
the player who can force a win would have to play very carefully to do so).

Sample: I won when trying my first 9x9. My moves (and computer's response):
B9 (H7)
D8 (I9)
C5 (H9)
E4 (G9)
F3 (I6)
A2 (loses)

My strategy is pretty much one of eliminating as many long rows/columns
as possible except along the edges; then when there are few such rows
left it's possible for me as a human to work out the whole game tree
(I can handle up to a depth of maybe 2 moves per player, tops).
That's why with each move I am destroying yet another full row and
another full column. I don't know why that should be an optimal strategy!

  One thing we humans can handle that the program can't, also, is
seeing when there's only one "major" arch left, and counting the number
of things that branch off of it. For example,

    OO.......
    OO.......
    OO.......
    OO.......
    .O.......
    .OOOOOOOO
    .OOOOOOOO
    OO.......
    .OOOOOOOO

That game board has one arch in the second column, and then eight independent "branches" coming off of it. We immediately determine that
the game will last exactly nine more moves (since each move knocks out
exactly one branch), and so we know who will win. But the computer
would have to examine (on the order of) 9-factorial game trees to
come to the same conclusion via alpha-beta!

What is abstractly a minor issue, but which human players may dislike,
is that your program is completely deterministic: it searches the
game tree in the same order all the time, and in particular takes the
rows and columns in order, so that it tends to respond with cubes
near the lower right. Some randomization would make it strike out into
other regions when there is no reason not to.

  Yeah, well, at the moment what I as a human player dislike is the fact
that the computer is so horribly easy to beat. Once it can win a few games
once in a while, it might be worth investing in a random number generator.
